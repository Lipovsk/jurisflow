# Flyway e migrations

## Estado atual

O backend possui as dependências do Flyway gerenciadas pelo Spring Boot e a migration `V1__baseline_schema_atual.sql`. A V1 foi derivada do schema físico capturado no backup aprovado de 20/07/2026 e foi validada em um PostgreSQL vazio e isolado.

A V1 reproduz as 9 tabelas de aplicação, 138 colunas, 62 constraints, 20 índices, 9 sequences, 9 IDs `GENERATED BY DEFAULT AS IDENTITY` e 11 foreign keys do schema aprovado. Ela não contém dados, correções de modelo ou alterações de regras de negócio.

## Proteções temporárias

Enquanto o banco principal ainda não tiver um baseline controlado:

- `spring.jpa.hibernate.ddl-auto=validate` impede o Hibernate de alterar o schema;
- `spring.flyway.enabled=false` impede aplicação automática de migrations pela configuração padrão;
- `spring.flyway.baseline-on-migrate=false` impede baseline implícito;
- `spring.flyway.clean-disabled=true` mantém o comando destrutivo desabilitado.

O profile `test` exige URL, usuário, senha e diretório de documentos próprios por variáveis de ambiente. Se essas variáveis não forem fornecidas, o teste falha sem usar a URL padrão do banco principal.

## Regra para a V1

`V1__baseline_schema_atual.sql` é destinada a bancos vazios. Ela não deve ser executada sobre o banco principal já existente, pois esse schema ainda não possui `flyway_schema_history`.

A integração do banco principal deve ocorrer em uma fase posterior, com backup validado, comparação final do schema e baseline explícito da versão 1. Somente depois dessa etapa o Flyway poderá ser habilitado nesse ambiente. Não manter `baseline-on-migrate=true` como configuração permanente e não usar `Flyway clean`.

## Validação realizada

Em 21/07/2026, no banco isolado `jurisflow_flyway_v1_test`:

- o primeiro startup aplicou V1 e passou pelo Hibernate `validate`;
- o segundo startup informou que o schema estava atualizado e não reaplicou V1;
- `flyway_schema_history` ficou com um único registro bem-sucedido da versão 1;
- o fingerprint estrutural coincidiu com os catálogos aprovados;
- todas as tabelas de aplicação permaneceram vazias;
- endpoints protegidos sem token retornaram HTTP 401;
- `mvn compile` e `mvn test` foram aprovados.

O banco principal `jurisflow` não foi alvo desses startups. O banco restaurado `jurisflow_restore_test` e os documentos temporários da validação anterior devem permanecer preservados.
